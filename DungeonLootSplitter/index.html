    Logic Explanation

    In Phase 3, I updated the Dungeon Loot Splitter so that the data does not disappear when the page refreshes. The program now saves the application state using localStorage. The state includes the loot array and the party size. Whenever the user adds or removes loot or changes the party size, the program updates the state in memory and then calls saveState(). The saveState() function creates an object that contains the loot array and party size and stores it in localStorage using JSON.stringify(). When the page loads, the restoreState() function runs first. It checks if saved data exists in localStorage and then converts the stored string back into a JavaScript object using JSON.parse(). The program validates the data and restores the loot items and party size into the application state. After the state is restored, updateUI() runs to display the loot list and totals on the page. It is important that updateUI() runs after restoreState() because the interface should always reflect the current state. If the UI updated before the state was restored, the page would appear empty even though data existed in storage. While debugging, I placed a breakpoint inside restoreState(). When the page loaded, execution paused before updateUI() ran. I could see that the loot array started empty and then items were added as the stored data was processed. After stepping through the code, updateUI() ran and the loot list appeared on the screen. This helped me understand the order of the application lifecycle. This phase helped demonstrate how JavaScript, HTML, and CSS work together to build a dynamic application that saves user input. It also reinforced concepts like arrays, objects, loops, conditionals, and functions while introducing JSON and localStorage for persistent data storage.
